## 2. Prohibited Activities

### 2.1 Illegal Activities
You may not use the Service for any illegal purpose, including but not limited to:

#### 2.1.1 Fraud and Deception
- **Fraudulent Schemes:** Pyramid schemes, Ponzi schemes, advance-fee fraud
- **Identity Theft:** Impersonating others or using false identities
- **Financial Crimes:** Money laundering, terrorist financing, tax evasion
- **Deceptive Practices:** False advertising, fake reviews, astroturfing

- **Export Control:** Violating export control laws or sanctions

- **Privacy Violations:** Unlawfully collecting or using personal data
- **Discrimination:** Discriminating based on race, religion, gender, sexual orientation, disability, age, or national origin

#### 2.1.3 Harmful Activities
- **Violence or Threats:** Threatening violence, inciting harm, promoting terrorism
- **Harassment:** Cyberbullying, stalking, doxing, unwanted sexual advances
- **Child Exploitation:** Child abuse, child sexual abuse material (CSAM), child exploitation

### 2.2 Abuse and Misuse
You may not:

#### 2.2.1 Abuse the Platform
- **Spam:** Sending unsolicited messages, spam, or bulk communications
- **Malicious Content:** Introducing viruses, malware, ransomware, or harmful code

#### 2.3.1 Harassment and Bullying

- **Doxing:** Publishing private or identifying information about others

#### 2.3.2 Discriminatory Content

- **Sexual Violence:** Content depicting sexual assault or sexual violence

You may not:

#### 2.4.1 Violate Privacy Rights

## 4. API Usage Rules

### 4.1 API Access
Our APIs (Application Programming Interfaces) allow developers to integrate with our Service. API access is subject to additional rules:

#### 4.1.1 Authentication
- **API Keys:** Use valid API keys obtained through our developer portal
- **Security:** Keep API keys secure and confidential
- **Rotation:** Regularly rotate API keys for security
- **Revocation:** Immediately revoke compromised keys

#### 4.1.2 Authentication Requirements
```
Authentication Headers:
- Authorization: Bearer YOUR_API_KEY
- X-API-Key: YOUR_API_KEY
```

### 4.2 Rate Limits
API usage is subject to rate limits to prevent abuse and ensure fair access:

#### 4.2.1 Free Tier
- **Requests:** 100 requests per hour
- **Daily Limit:** 1,000 requests per day
- **Burst Limit:** 10 requests per second

#### 4.2.2 Paid Tier
- **Requests:** 1,000 requests per hour
- **Daily Limit:** 10,000 requests per day
- **Burst Limit:** 100 requests per second
- **Enterprise:** Custom limits available

#### 4.2.3 Rate Limit Headers
```
Rate Limit Response Headers:
- X-RateLimit-Limit: Maximum requests
- X-RateLimit-Remaining: Remaining requests
- X-RateLimit-Reset: Time when limit resets
```

#### 4.2.4 Handling Rate Limits
When you hit rate limits:
- **Status Code:** 429 Too Many Requests
- **Retry-After:** Number of seconds to wait before retrying
- **Best Practice:** Implement exponential backoff

### 4.3 API Usage Guidelines

#### 4.3.1 Best Practices
- **Caching:** Cache API responses where appropriate
- **Batching:** Batch requests to reduce API calls
- **Webhooks:** Use webhooks instead of polling where available
- **Error Handling:** Implement proper error handling and retry logic

#### 4.3.2 Prohibited API Usage
You may not:
- **Exceed Rate Limits:** Circumvent rate limits or attempt to bypass them
- **Misuse APIs:** Use APIs in a way not intended or documented
- **Resell Access:** Resell API access without authorization
- **Reverse Engineer:** Attempt to reverse engineer APIs or extract data
- **Cache Inappropriately:** Cache sensitive or frequently changing data

#### 4.3.3 Data Usage
- **Own Data:** Only use API to access your own data or data you have rights to access
- **Data Ownership:** We retain ownership of aggregated and anonymized data
- **No Scraping:** Do not use APIs for scraping or harvesting data
- **License Restrictions:** Comply with all license restrictions

### 4.4 API Termination
We may suspend or terminate API access if:
- **Policy Violation:** You violate this policy or our Terms of Service
- **Abuse:** You abuse or misuse the APIs
- **Security Concerns:** Your API use poses security risks
- **Payment Issues:** Your subscription is past due or cancelled

## 6. Security Requirements

### 6.1 Account Security
You are responsible for maintaining the security of your account:

#### 6.1.1 Password Requirements
- **Strength:** Use strong, unique passwords
- **No Sharing:** Do not share your password with others
- **No Reuse:** Do not reuse passwords across multiple sites
- **Regular Updates:** Regularly update your password

#### 6.1.2 Multi-Factor Authentication (MFA)
We strongly recommend enabling MFA:
- **Available:** SMS, authenticator app, hardware key
- **Recommended:** Authenticator app or hardware key (more secure)
- **Backup Codes:** Save backup codes in a secure location

#### 6.1.3 Session Management
- **Sign Out:** Sign out when finished using the Service, especially on shared devices
- **Remember Me:** Use "remember me" only on personal, secure devices
- **Active Sessions:** Review and manage active sessions in account settings

### 6.2 Security Best Practices

#### 6.2.1 Data Security
- **Sensitive Data:** Be cautious when uploading sensitive or confidential data
- **Encryption:** Our Service encrypts data in transit and at rest
- **Access Control:** Only share access with trusted individuals
- **Backups:** Maintain your own backups of important data

#### 6.2.2 API Security
- **API Keys:** Treat API keys like passwords
- **Environment Variables:** Store API keys in environment variables
- **No Hardcoding:** Do not hardcode API keys in code or version control
- **HTTPS:** Always use HTTPS when making API calls

#### 6.2.3 Integration Security
- **Third-Party Services:** Be cautious when granting access to third-party services
- **OAuth:** Revoke access to applications you no longer use
- **Permissions:** Only grant necessary permissions
- **Regular Review:** Regularly review and revoke unnecessary access

### 6.3 Vulnerability Reporting
If you discover a security vulnerability:

#### 6.3.1 Responsible Disclosure
- **Report:** Report the vulnerability to us before disclosing publicly
- **Details:** Provide sufficient details to reproduce the vulnerability
- **Wait:** Allow us time to fix the vulnerability before disclosing
- **No Exploitation:** Do not exploit the vulnerability for any purpose

#### 6.3.2 Reporting Security Issues
**Email:** security@spreadsheetmoment.com
**Subject Line:** Security Vulnerability Report
**Required Information:**
- Description of the vulnerability
- Steps to reproduce
- Proof of concept (if applicable)
- Your contact information

#### 6.3.3 Safe Harbor
We commit to:
- **No Legal Action:** We will not pursue legal action if you follow responsible disclosure
- **Credit:** We will credit you for the discovery (if desired)
- **Bug Bounty:** We may offer a bug bounty for significant vulnerabilities

**Severe Violation:**

- Accidental policy violations
- First-time minor violations

#### 9.2.2 Moderate Violations

#### 9.2.3 Severe Violations

#### 9.3.2 Investigation

#### 9.3.3 Action

**Name:** [Designated Agent Name]

**Address:** [Physical Address]
**Phone:** [Phone Number]

#### 10.4.2 DMCA Notice Requirements
To be effective, a DMCA notice must include:

1. **Signature:** Physical or electronic signature of copyright owner or authorized agent
2. **Identification:** Identification of the copyrighted work claimed to have been infringed

5. **Statement:** A statement that you consent to the jurisdiction of the federal district court for the judicial district in which your address is located, or if your address is outside the United States, for any judicial district in which we may be found
