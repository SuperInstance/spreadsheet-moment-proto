## 1. Brand Architecture & Terminology

### 1.1 Core Brand Structure

**SuperInstance.AI** (Company/Brand)
- **Usage:** Always use "SuperInstance.AI" when referring to the company
- **Format:** Capital S, capital I, .AI suffix
- **Example:** "SuperInstance.AI builds POLLN - the spreadsheet AI that learns"

**POLLN** (Product/Technology)
- **Usage:** Pattern-Organized Large Language Network
- **Format:** All caps, no periods
- **Definition:** "A distributed intelligence system with living spreadsheet cells"
- **Example:** "POLLN learns your spreadsheet patterns and creates reusable AI agents"

### 1.2 Relationship Statement

**Standard Format:**
> "SuperInstance.AI builds POLLN - the spreadsheet AI that learns."

**Alternative Formats:**
- "POLLN by SuperInstance.AI"
- "SuperInstance.AI's POLLN technology"
- "The POLLN system from SuperInstance.AI"

### 1.3 Key Terminology

**Always Use:**
- Spreadsheet AI (not "AI for spreadsheets")
- Persistent learning (not "continuous learning")
- Reusable agents (not "repeatable automations")
- Universal cell architecture
- Rate-based change mechanics
- Confidence cascade
- Tile algebra

**Avoid:**
- AI assistant (too generic)
- Automation tool (too broad)
- Smart spreadsheets (vague)
- AI-powered (overused)

## 2. Voice & Tone

### 2.1 Brand Voice

**Expert but Accessible**
- Confident in technical depth
- Clear in explanations
- Patient with complex concepts

**Innovative but Practical**
- Forward-thinking about AI possibilities
- Grounded in real user problems
- Focused on tangible benefits

**Professional but Human**
- Business-appropriate language
- Conversational where helpful
- Empathetic to user challenges

### 2.2 Tone Variations

| Context | Tone | Example |
|---------|------|---------|
| **Marketing Website** | Confident, benefit-focused | "Turn spreadsheet repetition into reusable AI agents" |
| **Technical Documentation** | Precise, instructional | "Configure the confidence threshold using the cascade API" |
| **Blog Posts** | Conversational, educational | "Let's explore how persistent learning actually works" |
| **Error Messages** | Helpful, actionable | "We couldn't parse that formula. Try using standard Excel syntax." |
| **Social Media** | Engaging, concise | "Tired of repeating the same AI prompts? POLLN remembers." |

### 2.3 Writing Principles

**1. Lead with Benefits**
- ❌ "POLLN uses rate-based change mechanics"
- ✅ "POLLN learns from your changes, so you don't have to repeat yourself"

**2. Explain Complex Concepts Simply**
- ❌ "The universal cell architecture enables polymorphic type inference"
- ✅ "Every cell can learn any task - from calculations to AI analysis"

**3. Use Active Voice**
- ❌ "The system is designed to learn from user interactions"
- ✅ "POLLN learns from how you work"

**4. Be Specific**
- ❌ "Save time with automation"
- ✅ "Save 5+ hours weekly on financial reporting"

```

[Subheadline]: Turn spreadsheet repetition into reusable AI agents
[Primary CTA]: Try Free
```

**Value Proposition Grid (3 items):**

```
2. [Icon] Universal Cells
   "Every cell learns any task"
   "From calculations to AI analysis"

3. [Icon] Confidence Cascade
   "Only activates when certain"
   "Reduces AI hallucinations"
```

### 3.2 Blog Post Template

**Title Format:**
- How [Benefit]: "How POLLN Saves 5+ Hours Weekly on Financial Reports"
- What/Why [Concept]: "What Is Persistent Learning and Why It Matters"
- Comparison: "POLLN vs. Microsoft Copilot: The Learning Difference"

**Structure:**
```
# [Engaging Title]

[1-2 sentence hook explaining the problem]

## The Challenge: [User pain point]

[Describe the common frustration]

## How POLLN Solves It

[Explain the solution with specific examples]

## Technical Deep Dive (Optional)

[For technical audience - can be collapsed]

## Getting Started

[Actionable next steps]

## Conclusion

[Summary and call to action]
```

### 3.3 Documentation Standards

**Getting Started Guides:**
- Assume zero knowledge
- Include screenshots/videos
- Provide copy-paste examples
- Link to next steps

**API Documentation:**
- Include request/response examples
- Document error codes
- Provide rate limits
- Include authentication details

**Tutorials:**
- Real-world use cases
- Step-by-step instructions
- Expected outcomes
- Troubleshooting tips

## 4. Visual & Formatting Guidelines

### 4.1 Headings Hierarchy

```
# H1: Page Title (One per page)
## H2: Major Sections
### H3: Subsections
#### H4: Minor points (use sparingly)
```

### 4.2 Lists & Bullets

**Use numbered lists for:**
- Step-by-step instructions
- Sequential processes
- Prioritized items

**Use bullet points for:**
- Feature lists
- Benefits
- Non-sequential items
- Examples

### 4.3 Code & Technical Content

**Inline Code:** `=POLLN_LEARN(A1:B10)`
**Code Blocks:**
```javascript
// JavaScript example
const agent = POLLN.createAgent({
  range: 'A1:B10',
  task: 'forecast_revenue'
});
```

**Terminology Formatting:**
- **Bold** for key terms on first mention
- *Italics* for emphasis (use sparingly)
- `Code format` for functions, parameters, commands

### 4.4 Links & Navigation

**Internal Links:**
- Use descriptive anchor text
- Link to related content
- Avoid "click here"

**External Links:**
- Open in new tab when leaving site
- Use nofollow for non-trusted sources
- Include [external] indicator if helpful

## 5. Audience-Specific Guidelines

### 5.1 Business Decision Makers

**Focus:**
- ROI and time savings
- Competitive advantage
- Risk reduction
- Team productivity

**Language:**
- Business outcomes
- Quantitative benefits
- Case studies
- Security/compliance

**Avoid:**
- Technical jargon
- Implementation details
- Mathematical proofs

### 5.2 Technical Developers

**Focus:**
- API capabilities
- Integration options
- Customization potential
- Performance characteristics

**Language:**
- Technical specifications
- Code examples
- Architecture diagrams
- Best practices

**Include:**
- System requirements
- Authentication methods
- Rate limits
- Error handling

### 5.3 End Users

**Focus:**
- Ease of use
- Immediate value
- Visual demonstrations
- Common use cases

**Language:**
- Simple instructions
- Before/after examples
- Video tutorials
- Template library

**Avoid:**
- Advanced configuration
- Technical limitations
- Complex terminology

### 5.4 Researchers/Academics

**Focus:**
- Novel contributions
- Mathematical foundations
- Research methodology
- Experimental results

**Language:**
- Formal academic style
- Citations and references
- Technical precision
- Limitations discussion

**Include:**
- White papers
- Mathematical proofs
- Experimental data
- Future research directions

## 6. Quality Standards & Review Process

### 6.1 Content Review Checklist

**Before Publishing:**
- [ ] Brand terminology correct (SuperInstance.AI vs POLLN)
- [ ] Voice and tone appropriate for audience
- [ ] Technical accuracy verified
- [ ] Links working and relevant
- [ ] No spelling/grammar errors
- [ ] Accessibility standards met
- [ ] SEO keywords included naturally
- [ ] Call-to-action clear and prominent

### 6.2 SEO Guidelines

**Title Tags:** 50-60 characters including brand
**Meta Descriptions:** 150-160 characters, benefit-focused
**Headings:** Include primary keywords naturally
**Image Alt Text:** Descriptive, include keywords if relevant
**Internal Linking:** 2-3 relevant internal links per page

### 6.3 Accessibility Standards

**Text:**
- Minimum contrast ratio 4.5:1
- Font size minimum 16px for body
- No text in images (use alt text)

**Structure:**
- Semantic HTML headings
- Descriptive link text
- ARIA labels for interactive elements

**Media:**
- Video transcripts
- Audio descriptions
- Captions for all video content
